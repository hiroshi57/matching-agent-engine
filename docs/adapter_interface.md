# コア/アダプタ境界 インターフェース定義 (draft v0.1)

> ステータス: **仕様書ドラフト（承認待ち）** / 対象: B4 matching-agent-engine / 作成: 2026-07-09

## 0. 設計原則

**コアエンジン（業種非依存）とアダプタ（業種依存）を厳密に分離**し、新業種への適用を
アダプタ実装のみ（コード変更なし・yaml＋インポータ）で2週間以内に可能にする。

## 1. アダプタが提供するもの（Adapter Interface）

```python
class Adapter(Protocol):
    slots: list[SlotDef]                 # 条件スロット定義(slots.yaml)
    def load_items(self) -> list[Item]:  # アイテム取込(CSV/API)
    tone: ToneConfig                     # 質問文・提案文のトーン
```

- `SlotDef`: `{name, type(enum/range/text), required, question}` 例(不動産): エリア/家賃/間取り/こだわり
- `Item`: `{id, attributes: dict}` — **id は実在アイテムの一意キー（必須）**

## 2. コアが提供するもの（業種非依存）

1. **対話マネージャ**: 発話→スロット抽出(構造化出力)→不足スロットの質問生成。1ターン1問・最大5ターンで候補提示。
2. **ハイブリッド検索**: スロット→構造化フィルタ(SQL)＋自由文→ベクトル検索(pgvector)→統合。
3. **実在IDバリデータ（差別化の核）**: 候補は**DBに実在するアイテムIDのみ**。
   LLMが候補を生成/改変した場合は破棄する。
4. **理由生成**: 候補ごとに「なぜ合うか」を、抽出スロットとアイテム属性の**対応のみ**から生成
   （属性にない情報を足さない／属性照合で検証）。
5. **イベントログ**: 発話・抽出スロット・提示候補・クリック・離脱を記録。

## 3. マーケティング・フィードバック（差別化）

対話ログを週次分析し、頻出ニーズ語彙／成約に効いた条件組合せ／候補ゼロ条件 を
広告のキーワード・クリエイティブ・LP改善示唆として出力（匿名セッション単位）。

## 4. 受け入れ基準

- 自動対話テストで**提示候補の実在ID率100%**（LLM創作候補の混入ゼロ）。
- 3ターン以内候補提示率90%以上。
- 理由文にアイテム属性にない情報が含まれないこと（属性照合テスト）。
- 不動産→求人へのアダプタ切替がコード変更なし（yaml＋インポータのみ）で動くこと。
